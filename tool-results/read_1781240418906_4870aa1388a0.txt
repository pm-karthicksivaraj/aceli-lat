    10→    10→## 1. Purpose
    11→    11→
    12→    12→This document defines the end-to-end sprint plan for building the full production application proposed in the Aceli Africa RFP. It is designed for execution by a multi-agent engineering organisation and covers delivery phases, AI-agent rules, architecture constraints, sprint goals, backlog themes, quality gates, documentation outputs, release governance, and rollout requirements.
    13→    13→
    14→    14→This plan is intentionally **production-first**, not MVP-first. Every sprint must produce working software, controlled documentation, test evidence, and operational artifacts.

    18→    18→## 2. RFP-aligned delivery constraints
    19→    19→
    20→    20→The build must comply with the following non-negotiable constraints from the RFP:
    21→    21→
    22→    22→- Salesforce remains the primary system of record for lender relationship data and relevant structured lender records.
    23→    23→- The LAT platform is the field-facing workflow and intelligence layer.
    24→    24→- Power BI remains the analytics and reporting layer; it must not become the operational system of record.
    25→    25→- The benchmarking data store must be fed through a controlled, validated pathway only.
    26→    26→- No shadow databases, unmanaged exports, or manual reconciliation routines are allowed.
    27→    27→- All AI-suggested updates require human review before write-back to approved systems.
    28→    28→- All AI-assisted decisions and outputs must produce audit-ready logs.
    29→    29→- No lender-level or borrower-level data may be used to train vendor-owned, shared, or third-party models.
    30→    30→- All processing must run only within Aceli-approved enterprise tenants or approved vendor platforms.
    31→    31→- The solution must support low-bandwidth operation and offline capture with later synchronization.
    32→    32→- The solution must target 99.5% availability during business hours.
    33→    33→- Training, change management, rollout support, and administrator handover are in scope.

    37→    37→## 3. Required and allowed technology stack
    38→    38→
    39→    39→Only the following stack elements are explicitly mandated or directly referenced by the RFP and must be treated as required or controlled:
    40→    40→
    41→    41→### 3.1 Required enterprise platforms
    42→    42→- **Salesforce**: system of record, API integration target, structured lender data authority.
    43→    43→- **Power BI**: reporting, analytics, and visualization layer.
    44→    44→- **Google Workspace**: approved enterprise tenant for collaboration/document operations where applicable.
    45→    45→- **Claude Team or Claude Enterprise**: approved AI platform option for governed AI processing.
    46→    46→
    47→    47→### 3.2 Approved-by-review components
    48→    48→Any additional application runtime, middleware, storage, queue, mobile/web framework, observability stack, transcription component, or integration platform must:
    49→    49→- pass Aceli information security review,
    50→    50→- be added to the approved list before contract execution if vendor-provided,
    51→    51→- operate without violating tenant, data residency, audit, and no-training requirements.
    52→    52→
    53→    53→### 3.3 Recommended implementation pattern
    54→    54→To stay faithful to the RFP, the AI agents must use this technology pattern:
    55→    55→- **Frontend channel**: mobile-first responsive web application or PWA for field use, with desktop-ready HQ experience.
    56→    56→- **Application services layer**: controlled API/service layer for workflow orchestration, review state, sync, audit, and exception handling.
    57→    57→- **AI services layer**: governed transcription, extraction, summarization, and prompt orchestration using approved enterprise AI tenants only.
    58→    58→- **Integration layer**: Salesforce APIs, controlled benchmarking feed, approved analytics outputs.
    59→    59→- **Observability layer**: immutable audit logs, application logs, sync logs, AI decision logs, user action logs.
    60→    60→- **Identity and access layer**: role-based access control aligned to country, HQ, executive, and product owner roles.
    61→    61→
    62→    62→### 3.4 Strict stack rules for AI agents
    63→    63→AI agents must not introduce any of the following without explicit approval:
    64→    64→- direct production dependency on unapproved public LLM endpoints,
    65→    65→- local shadow databases acting as source of truth,
    66→    66→- spreadsheet export-based reconciliation workflows,
    67→    67→- unreviewed schema mutations in Salesforce,
    68→    68→- dashboards used as transactional capture tools,
    69→    69→- AI auto-writeback bypassing reviewer approval.

    73→    73→## 4. AI agent operating model
    74→    74→
    75→    75→### 4.1 Core agent roles
    76→    76→- Delivery Orchestrator Agent
    77→    77→- Product Owner Agent
    78→    78→- Business Analyst Agent
    79→    79→- Solution Architect Agent
    80→    80→- Salesforce Integration Agent
    81→    81→- AI/ML Workflow Agent
    82→    82→- Frontend Engineering Agent
    83→    83→- Backend/API Engineering Agent
    84→    84→- Data Migration Agent
    85→    85→- DevSecOps Agent
    86→    86→- QA Automation Agent
    87→    87→- Technical Writer Agent
    88→    88→- Release Manager Agent
    89→    89→- Change Management Agent
    90→    90→- Red Team / Compliance Agent
    91→    91→
    92→    92→### 4.2 Mandatory agent rules
    93→    93→Every AI agent must follow these rules:
    94→    94→
    95→    95→1. **Documentation-first rule**: no feature build starts without linked requirement IDs.
    96→    96→2. **Traceability rule**: every code task maps to SRS/FRS/user story/API contract/test case IDs.
    97→    97→3. **Approval-gate rule**: any Salesforce write-back logic requires architect + QA + compliance approval.
    98→    98→4. **No hallucination rule**: if a platform detail is unknown, create an assumption record; do not invent.
    99→    99→5. **Human review rule**: all AI-generated lender updates must stop at reviewer state before commit.
   100→   100→6. **No training rule**: no protected data enters training or fine-tuning workflows.
   101→   101→7. **Security-first rule**: secrets, tokens, credentials, and tenant settings must never be hardcoded.
   102→   102→8. **Migration discipline rule**: no production data migration without dry run, reconciliation report, and rollback plan.
   103→   103→9. **Testing rule**: no story is complete without automated tests or explicit approved test evidence.
   104→   104→10. **Audit rule**: all meaningful user and AI actions must generate structured logs.
   105→   105→11. **Offline-first rule**: field features must degrade gracefully under weak or zero connectivity.
   106→   106→12. **Definition-of-done rule**: code, tests, docs, security checks, and release notes are all required.
   107→   107→13. **Post-coder validation rule**: all generated code must run through linting, static analysis, unit/integration checks, and architectural policy validation.
   108→   108→14. **No repetition rule**: agents must reuse approved modules, templates, and patterns; avoid duplicate implementations.
   109→   109→15. **Release governance rule**: no production release without signed release checklist and rollback plan.
   110→   110→
   111→   111→### 4.3 Agent confidence and HITL thresholds
   112→   112→- Confidence > 0.90: autonomous execution allowed within approved scope.
   113→   113→- Confidence 0.70–0.89: autonomous draft allowed, mandatory peer-agent review required.
   114→   114→- Confidence < 0.70: stop and escalate to HITL or lead architect review.
   115→   115→- Any security, data governance, or production data task: HITL review always mandatory.

   188→   188→## 6. Delivery phases and sprint calendar
   189→   189→
   190→   190→The sprint structure below aligns with the RFP delivery model:
   191→   191→- **Phase 0 (Weeks 1–4)**: discovery, baseline capture, design sprint, architecture confirmation.
   192→   192→- **Phase 1 (Weeks 5–13)**: pilot build and deployment in one country by Day 90.
   193→   193→- **Phase 2 (Months 4–6)**: phased rollout to remaining four countries.
   194→   194→- **Phase 3 (Post-rollout)**: warranty, support, continuous improvement, operational review.
   195→   195→
   196→   196→### Sprint cadence
   197→   197→- Sprint length: 2 weeks, except Sprint 0 and Sprint 6 which may be shorter/longer to align with pilot deadline.
   198→   198→- Daily AI sync: automated status, blocker, dependency, and quality update.
   199→   199→- Weekly governance review: compliance, scope, architecture, and delivery risk review.
   200→   200→- End-of-sprint outputs: working increment, evidence pack, updated docs, release notes.
